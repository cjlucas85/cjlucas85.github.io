Shell Scripting is terrible, and you need to stop using it.

For years, I’ve encountered a fair amount of shell scripting being used on various projects. Often shell scripting finds its away in tasks related to build systems, automated testing, source code management and similar tasks. These are all areas which well trained Software Engineers with good intentions have deemed a suitable tool for the job. They’re wrong. Instead of making things more streamline, a shell script will make things more unpredictable. It will not be portable. It will widen the scope of knowledge required to maintain the project. All of this at the price of little gain over a conventional programming language.

In this article, I will demonstrate the problems encountered by shell scripts followed by alternatives.

I will use MS Batch File as a demonstration for problems; although, keep in mind that BASH Script, PowerShell, and others have similar problems.

Syntax Errors

In a general programming language, you get a fair amount of syntax checking. This syntax checking comes with it some useful tools to addressing the offending syntax error.

You’ll be lucky to have any of that with a Shell Script.

Instead, a shell script is typically feeding lines into the command interpeter. Errors that occur may or may not halt the execution of the script. Sometimes, you’ll get useful information.

To demonstrate this, I’ll use a simple IF ELSE script:

@echo off
set X=10
echo "Before blocks"
if "%X%" == "10" (
    echo "X is 10"
) else (
    echo "X is not 10"
)
echo "After blocks"

This works as expected.

"Before blocks"
"X is not 10"
"After blocks"

If we alter this so that “else” is on its own line, the command line interpreter will treat “else” on it’s own like a function, system command or application:

@echo off
set X=10
echo "Before blocks"
if "%X%" == "10" (
    echo "X is 10"
) 
else (
    echo "X is not 10"
)
echo "After blocks"

In this case it happens to continue theh operation while showing noticeable signs of failure.

"Before blocks"
"X is 10"
'else' is not recognized as an internal or external command,
operable program or batch file.
"X is not 10"
"After blocks"

If the parenthesis after else is continued to the next line, the CMD.exe is able to detect this as a syntax error, and it will exit the entire script:

@echo off
set X=10
echo "Before blocks"
if "%X%" == "10" (
    echo "X is 10"
) else 
(
    echo "X is not 10"
)
echo "After blocks"

In this case, a syntax error is encountered and the script exits.

"Before blocks"
The syntax of the command is incorrect.

There are two conclusions from this example:

  1. Syntax Errors sometimes exit, sometimes continue.
  2. Syntax is very strict.

This combination makes Shell Scripting tedius. Without printing, how were we to determine whether the IF ELSE was constructed properly? It’s one thing to have strict syntax, it’s another for the shell to not have predicable methods for syntax errors. With shell scripting languages such as BASH, the line number may be printed along with a helpful message, but that is if the shell catches it and doesn’t try to interpret it as something else.

Unpredictable outcomes are not just a matter of syntax but also runtime errors.

Runtime Errors

Modern programming languages are equiped with exception handling or other error handling mechanisms. Often these are errors which will indicate to other parts of the code that an operation could not be completed. If this failure isn’t checked, it may even halt the program all together. When this occurs a useful text with a description of the error and a trail of function calls are listed. This is often called a stack trace. An alternative to using a stack trace is using a debugger. Debuggers typically run or run along side of the program. They give useful information about the state of the application along with errors encountered.

You’ll be lucky if the script exits on errors or reports line numbers.

Instead, errors will be ignored by default. The reason is because Shell Scripting is essentially the process of calling other applications with a few built-in shell commands mixed together. The interface between applications and thet shell is either through streams (stdout & stderr) and return codes. Since text streams can be hard to parse for errors, the most reliable method of determining successful execution of a program is through the return code. This means the caller, the script, must check the values that are returned by the callee, the application the script calls.

I’ll demonstrate the problem with a simple script. It will contain a routine called “my_func” which takes two paraeters and prints them to the screen. The script will call “my_func” twice with a different pair of parameters:

@echo off
call :my_func, "A", "B"
call :my_func, "C", "D"
exit /B %ERRORLEVEL%

:my_func
echo %~1 %~2
EXIT /B 0

This will produce the following output.

A B
C D

Now, let’s throw a wrench at this by adding “create_problem_here” in our function:

@echo off
call :my_func, "A", "B"
call :my_func, "C", "D"
exit /B %ERRORLEVEL%

:my_func
echo %~1 %~2
create_problem_here
EXIT /B 0

And when we run this:

A B
'create_problem_here' is not recognized as an internal or external command,
operable program or batch file.
C D
'create_problem_here' is not recognized as an internal or external command,
operable program or batch file.

Here we see the problem encountered twice, and yet the application continued to run. This is because CMD.exe places the return code in the variable “errorlevel” and leaves checking the status to the script to handle. Alternatively, shell commands can be given logical AND and logical OR operations. These operations can provide some error checking and mitigation. We can fix the code to give us something close to what we expect; exit on first error of “create_problem_here”:

@echo off
call :my_func, "A", "B"
call :my_func, "C", "D"
exit /B %ERRORLEVEL%

:my_func
echo %~1 %~2
create_problem_here || echo whoops! && call :halt
EXIT /B 0

:halt
call :haltHelper 2> nul

:haltHelper
() 
exit /b

Here we use some short circuit evaluation. If “create_problem_here” does not execute successfully, it will print “whoops!” and halt the program; otherwise, it will continue.

Since “create_problem_here” isn’t an internal or external command, we get the following:

A B
'create_problem_here' is not recognized as an internal or external command,
operable program or batch file.
whoops!

While it appears we’ve solved a problem, we’ve actually exposed another. We checked the one line for an error; however, all other lines have not been checked. This creates a dilemma and huge gap for determination. The delimma presented is where to check for errors. Naturally, you can be pedantic and check everything for errors, but this will pad the code with a litany of checks. On the otherhand, you could just check code that is critical for success; afterall, some builtin commands like echo will not likely fail or fail critical to the rest to other operations. If errors are left unchecked, the script potentially becomes indeterminant. Scripts often rely on certain state to be in the right place. This can be in the form of the present working directory, files, folders, environment variables, and even commands available on the system. If any of these things are missing or modified in a way that wasn’t anticipated, the script’s execution is going to be misaligned and faulty. This leads to unpredictable behavior.

This means that scripts must be written in a pedantic fashion to account for the multiple failure points; however, Shell Scripts are rarely engineered. Frequently, a shell script is cobbled together from a set of commands that worked, and they have been placed into a file for reuse. It’s often a distant afterthought to include any error handling.

Why Do Developers Choose Shell Scripts?

Before any solutions can be presented, we need to discuss what motivations developers have for using shell scripts.

Mostly

Simply put, developers often form shell scripts from the commands they invoke on the command line to get tasks done. The simplest solution to recreating that success and using it over and over again is to pack that into a shell script file. This is why it’s not as simple as saying “Use a general programming language.”

I’ll illustrate this by making a script which will check out a project and compile it. If the project already exists, it will delete the directory before checking out. Try not to over think why we might want this, concentrate on the idea that there are potential failure points in the implementation. Naturally the composition of commands will look like this:

rm -rf project
svn co svn://svn.company.com/project
cd project
make
cd ..

Naturally this can be packed into a shell script without any motification. This application looks much different in Python. The following is a simplistic conversion of the same tasks:

import os
os.system('rm -rf project')
os.system('svn co svn://svn.company.com/project')
os.chdir('project')
os.system('make')
os.chdir('..')

This required one more line and more typing, but did we gain anything from it? Well, actually there is at least one perk. The call to “os.chdir(‘project’)” will raise an exception if it cannot change directory whereas the script above will continue execution (unless the shell provides exit on error). In fact, the more “os.system” calls are converted into Python equivilents, the more error checking is provided. Let’s put a bit more care into this script:

import os
import sys
cwd = os.getcwd()
# REMOVE PROJECT IF IT EXISTS
try:
    os.rmdir('project')
except FileNotFoundError:
    pass
# CHECK OUT PROJECT
if os.system('svn co svn://svn.company.com/project') != 0:
    print('Could not check out repository', file=sys.stderr)
    sys.exit(1)
# ENTER PROJECT DIRECTORY
os.chdir('project')
# MAKE PROJECT
if os.system('make') != 0:
    print('Compile Failed', file=sys.stderr)
    os.chdir(cwd)
    sys.exit(1)
# RETURN TO ORIGINAL DIRECTORY
os.chdir(cwd)

Admittedly, this is lengthy. It required more consideration. It’s not likely the first solution someone would use to approach this problem. However, this is far more reliable than the first script presented. Error checking is done for things we anticipate. Exceptions will raise for the errors we didn’t anticipate. If the choice is to be made of which script is to be used for long term development, it’s clear which oneshould be used.

What does this all mean?

Using shell scripting is often a path of convenience. Switching languages may provide error checking without much effort; however, propper error checking will require some careful consideration and effort.

Sometimes

At times there isn’t any choice in the matter.

Evironment Restrictions: Certain things are not available because of the scope of the system or the environment. When installing a program on Windows, the only scripting languages you can rely on are MS Batch File, VBScript and PowerShell. Linux will at least have /bin/sh available on even the most minimalistic distribution, and it may be your project’s peragative to keep the OS installation image small.

Organizational Restrictions: Individuals in the organization do not want languages they are not famailiar with populating their poject.

When you can use an alternative to shell scripts; use it.

Alternatives

Python and Ruby are natural alternatives to shell scripting. Both are easy to learn, have support from a plethera of third party modules, and their communities are large. Chances are if you’re apart of a project, there may already exist a bias toward using either one. If that’s the case, that is the path toward replacing shell scripts from the project. Otherwise, my recommendation is to use Python. Python has slightly more reach with utilities and application development than Ruby, and the language isn’t typecast to a single framework.
